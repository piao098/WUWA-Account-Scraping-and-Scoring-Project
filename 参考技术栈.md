# 参考技术栈

本文档提炼 `G:\code base\cangbaoge` 藏宝阁项目中可复用的技术思路，并改写为“螃蟹鸣潮账号评分器”的参考方案。目标不是照搬代码，而是复用成熟做法：先稳定拿数据，再标准化，再评分，再筛选和自动化。

## 1. 数据入口逆向

参考藏宝阁项目的 `API_REFERENCE.md`，本项目也应先明确螃蟹网站的数据入口，而不是一上来写复杂爬虫。目标是找到鸣潮账号列表页、分页参数、详情页地址、搜索/筛选参数，以及是否存在可直接返回 JSON 的接口。

量化目标：完成 `API_REFERENCE.md`，记录至少 `1` 个列表入口、`1` 个详情入口、`5` 个核心字段来源；能稳定获取不少于 `20` 条鸣潮账号列表数据。

## 2. 列表抓取策略

参考藏宝阁项目的 `auto_fetch.py`：优先使用轻量 HTTP 请求抓列表，如果接口需要浏览器上下文，再使用 Selenium 或真实浏览器登录态兜底。抓取时保留分页、区服、价格、标题、发布时间、详情链接等基础字段。

量化目标：一次运行至少抓取 `100` 个账号列表项；失败请求自动重试 `2-3` 次；重复账号按详情链接或账号 ID 去重，重复率控制到 `0`。

## 3. 浏览器兜底机制

参考藏宝阁项目的 `live_browser_fetch.py`：当普通请求拿不到数据时，启动 Chrome/Edge 的真实用户 profile，通过 Selenium 附加到浏览器调试端口执行页面内请求。这个方式适合处理需要登录、Cookie、动态参数或前端渲染的网站。

量化目标：保留 `HTTP 抓取` 和 `浏览器抓取` 两条路径；当 HTTP 抓取失败时，浏览器路径至少能成功解析 `10` 个详情页中的 `8` 个。

## 4. 反风控与限速

参考藏宝阁项目中的翻页延迟、服务器延迟和 cooldown 文件机制。螃蟹项目也需要避免高频访问：分页之间加固定或随机延迟，遇到验证码、系统繁忙、访问拒绝时写入冷却状态，等待后再重试。

量化目标：默认请求间隔不少于 `2` 秒；连续失败 `3` 次后进入冷却；所有风控类失败写入日志，并记录恢复时间。

## 5. 详情数据解析

参考藏宝阁项目“列表只拿基础信息，详情才拿评分资产”的设计。本项目应从详情页中解析鸣潮账号核心资产：角色、共鸣链、武器、精炼、联觉等级、数据坞、星声/抽数、已养成队伍、安全说明和截图文本。

量化目标：随机抽取 `10` 个详情页，至少 `8` 个能解析出角色、武器、价格、区服和安全说明；核心字段缺失率控制在 `20%` 以内。

## 6. 数据标准化

参考藏宝阁项目的 `score_results.json` 输出结构。螃蟹项目应将网页原始字段转换成统一 JSON，避免后续评分逻辑依赖网页格式。建议保留 `raw` 字段，便于排查解析错误。

量化目标：每个账号统一输出 `account_id`、`price`、`title`、`server`、`characters`、`weapons`、`resources`、`risk_flags`、`detail_url`、`raw`；批量数据能保存为 `data/accounts_raw.json`。

## 7. 游戏资产配置表

参考藏宝阁项目从 `game_auto_config.js` 中提取名称和品质映射的思路。鸣潮项目应建立独立配置表，维护角色、武器、版本强度、限定属性、关键命座和推荐权重。

量化目标：完成 `configs/characters.json`、`configs/weapons.json`、`configs/scoring_rules.json`；至少覆盖 `30` 个角色、`30` 把武器和 `5` 类评分维度。

## 8. 评分模型

参考藏宝阁项目的分项评分模式：芯片、核芯、配件、装备、附加项分别计分。鸣潮项目可拆成角色价值、武器价值、队伍成型度、资源进度、安全风险和性价比。总分建议先设为 `100` 分，后续再按实际市场结果调整。

量化目标：每个账号输出 `character_score`、`weapon_score`、`team_score`、`resource_score`、`risk_score`、`total_score`；评分结果能解释每一项加分和扣分原因。

## 9. 性价比模型

参考藏宝阁项目的“总分 - 价格期望分”思路。鸣潮账号不能只按总分排序，还要结合价格判断是否低估。可以先用简单公式：同价位内排序、单位价格得分、指定角色溢价、风险扣分。

量化目标：每个账号生成 `value_score`；按 `0-100`、`100-300`、`300-600`、`600+` 价格段分别输出 Top `5`；报告中同时展示总分排名和性价比排名。

## 10. 批量评分与报告

参考藏宝阁项目的 `batch_scorer.py` 和 `notify.py`。本项目应支持批量读取原始账号数据，逐个评分，生成 JSON 和可读报告。报告优先做 Markdown 或 HTML，展示价格、核心角色、核心武器、推荐理由和风险提示。

量化目标：一次运行处理不少于 `100` 个账号；生成 `data/score_results.json` 和 `reports/report.html`；报告展示 Top `20` 推荐账号。

## 11. 通知机制

参考藏宝阁项目的 PushPlus 推送。螃蟹项目后期可以加入 PushPlus、Bark、企业微信或邮件，只推送高性价比账号，避免信息过载。

量化目标：当账号 `total_score >= 75` 或 `value_score >= 80` 时触发通知；每轮最多推送 `10` 个账号；推送内容包含价格、核心资产、评分和详情链接。

## 12. 交互筛选面板

参考藏宝阁项目的 `interactive_filter_server.py`。本项目可以先不做复杂前端，只做一个本地 HTTP 面板，读取已有评分结果，按价格、角色、武器、风险项、性价比重新筛选。

量化目标：提供 `/panel`、`/apply`、`/health` 三个入口；筛选时不重新抓取网页，只读取 `score_results.json`；筛选结果能在 `1` 秒内返回。

## 13. 一键运行管线

参考藏宝阁项目的 `run_all.py`。本项目最终应把抓列表、抓详情、评分、报告、通知串成一个入口，并加锁防止重复运行。

建议流程：

```text
fetch_list.py -> fetch_detail.py -> batch_scorer.py -> report.py -> notify.py
```

量化目标：执行 `python run_all.py` 能完成全流程；生成日志 `logs/run_all.log`；失败账号写入 `data/errors.json`；重复运行不会并发冲突。

## 14. 本地文件与敏感信息管理

参考藏宝阁项目的 `.gitignore`。螃蟹项目需要把 Cookie、Token、浏览器 profile、日志、截图、原始抓取数据和评分结果区分管理。代码可以提交，运行数据默认不提交。

量化目标：建立 `.gitignore`；忽略 `data/`、`logs/`、`*.cookie`、`*_cookies.json`、`notify_config.json`、浏览器 profile、截图和临时文件。

## 15. 环境管理

遵循本项目 `AGENTS.md`：使用 `uv` 仅做环境管理，包括创建虚拟环境、安装依赖和运行命令；不把项目改成 uv 项目，除非明确需要。

量化目标：提供 `requirements.txt`；用 `uv venv` 创建环境，用 `uv pip install -r requirements.txt` 安装依赖，用 `uv run python run_all.py` 运行流程。
